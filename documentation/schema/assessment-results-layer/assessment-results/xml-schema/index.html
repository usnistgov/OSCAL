Assessment Results Model XML Schema Reference

Schema download oscal_assessment-results_schema.xsd

Schema version 1.0.0-milestone3

Short name oscal-ar

XML namespace http://csrc.nist.gov/ns/oscal/1.0

The OSCAL assessment results format is used to describe the information typically provided by an assessor following an assessment.

The root of the OSCAL assessment results format is assessment-results.

address

Name Address

Description A postal address.

This element appears inside: location and party.

The address element has the following contents (in order):

  • type attribute (string) [optional] Indicates the type of address.

    The value may be locally defined, or one of the following:

    • home: A home address.
    • work: A work address.

  • addr-line elements (string) [0 to ∞] A single line of an address.

  • city element (string) [0 or 1] City, town or geographical region for mailing address

  • state element (string) [0 or 1] State, province or analogous geographical region for mailing address

  • postal-code element (string) [0 or 1] Postal or ZIP code for mailing address

  • country element (string) [0 or 1] Country for mailing address

algorithm

Name Hash algorithm

Description Method by which a hash is derived

An attribute (string)

The value may be locally defined, or one of the following:

  • SHA-224
  • SHA-256
  • SHA-384
  • SHA-512
  • SHA3-224
  • SHA3-256
  • SHA3-384
  • SHA3-512

This attribute appears on: hash.

Remarks

Valid values MUST be a value defined in the W3C XML Security Algorithm Cross-Reference Digest Methods (W3C, April 2013).

annotation

Name Annotation

Description A name/value pair with optional explanatory remarks.

This element appears inside: location, party, role, responsible-party, user, component, inventory-item, implemented-component, responsible-role, objectives, controls, control-objectives, objective, method, include-subject, exclude-subject, assets, test-method, task, include-activity, exclude-activity, results, finding, observation, relevant-evidence, risk, tracking-entry, required, and remediation.

The annotation element has the following contents:

  • name attribute (NCName) [required]

  • id attribute (NCName) [optional]

  • ns attribute (string) [optional]

  • value attribute (string) [optional] Indicates the value of the characteristic.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

Annotations permit the deployment and management of arbitrary controlled values for any purpose useful to an application or implementation scenario.

The value (flag) of an annotation is a simple scalar value or list of values. The body of the annotation offers remarks in explanation of or reference to this value.

assessment-activities

Name Assessment Activities

Description Identifies the assessment activities and schedule. In the assessment plan, these are planned activities. In the assessment results, these are the actual activities performed.

This element appears inside: assessment-results.

The assessment-activities element has the following contents (in order):

  • test-method elements [0 to ∞] Identifies an individual test method.

  • schedule element [0 or 1] Identifies the schedule for the assessment activities.

  • include-activity elements [0 to ∞] Identifies an assessment activity. In the assessment plan, this is an intended/in-scope activity. In the assessment results, this identifies an activity that was actually performed.

  • exclude-activity elements [0 to ∞] Identifies an activity explicitly excluded from the assessment. In the assessment plan, this clarifies activities that are out-of-scope or prohibited. In the assessment results, this could be used to explicitly identify an activity that was planned, but not performed.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

assessment-results

Name Security Assessment Results (SAR)

Description Security assessment results, such as those provided by a FedRAMP assessor in the FedRAMP Security Assessment Report.

assessment-results is the root (containing) element of this schema.

The assessment-results element has the following contents (in order):

  • uuid attribute (uuid) [required]

  • metadata element [1] Provides information about the publication and availability of the containing document.

  • import-ap element [1] Used by the SAR to import information about the original plan for assessing the system.

  • objectives element [1] Identifies the controls and control being assessed and their control objectives. In the assessment plans, these are the planned controls and objectives. In the assessment results, these are the actual controls and objectives, and reflects any changes from the plan.

  • assessment-subjects element [0 or 1] Identifies system elements being assessed, such as components, inventory items, and locations. In the assessment plan, this identifies the planned assessment subject. In the assessment results this is the actual assessment subject, and reflects any changes from the plan.

  • assets element [0 or 1] Identifies the assets used to perform this assessment, such as the assessment team, scanning tools, and assumptions.

  • assessment-activities element [0 or 1] Identifies the assessment activities and schedule. In the assessment plan, these are planned activities. In the assessment results, these are the actual activities performed.

  • results elements [1 to ∞] Used by the assessment results and POA&M. In the assessment results, this identifies all of the assessment observations and findings, initial and residual risks, deviations, and disposition. In the POA&M, this identifies initial and residual risks, deviations, and disposition.

  • back-matter element [0 or 1] A collection of citations and resource references.

assessment-subjects

Name Subject of Assessment

Description Identifies system elements being assessed, such as components, inventory items, and locations. In the assessment plan, this identifies the planned assessment subject. In the assessment results this is the actual assessment subject, and reflects any changes from the plan.

This element appears inside: assessment-results.

The assessment-subjects element has the following contents (in order):

  • include-subject elements [1 to ∞] Identifies exactly what will be the focus of this assessment. Anything not explicitly defined is out-of-scope.

  • exclude-subject elements [0 to ∞] Identifies what is explicitly excluded from this assessment. Used to remove a subset of items from groups of explicitly included items. Also used to explicitly clarify off-limit items, such as hosts to avoid scanning.

  • local-definitions element [0 or 1] Allows control objectives, users, components, and inventory-items to be defined within the assessment plan or assessment results for circumstances where they are not appropriately defined in the SSP. NOTE: Use the assessment plan or assessment results metadata to define additional locations if needed.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

back-matter

Name Back matter

Description A collection of citations and resource references.

This element appears inside: assessment-results.

The back-matter element has the following contents:

  • resource elements [0 to ∞] A resource associated with the present document, which may be a pointer to other data or a citation.

Remarks

Provides a collection of identified citation and resource objects that can be referenced by a link with a rel value of "reference" and an href value that is a fragment "#" followed by a reference to a citation identifier. A citation can reference a resource by specifying a local target value using the same fragment-identifier approach.

The following is a contrived example to show the use of link, citation, and resource.

<profile>
<metadata>
<link rel="citation" href="#resource1">My citation</link></metadata>
<back-matter>
<resource id="resource1">
<rlink href="https://example.org/some-resource"></rlink></resource></back-matter></profile>

component

Name Component

Description A defined component that can be part of an implemented system.

This element appears inside: tools and local-definitions.

The component element has the following contents (in order):

  • uuid attribute (uuid) [required] The unique identifier for the component.

  • component-type attribute (string) [required]

    The value may be locally defined, or one of the following:

    • software: Any software, operating system, or firmware
    • hardware
    • service: A service that may provide APIs.
    • interconnection: A connection to something outside this system.
    • policy
    • process
    • procedure
    • plan
    • guidance: Any guideline or recommendation
    • standard
    • validation: An external assessment performed on some other component, that has been validated by a third-party.

  • title element (markup-line) [1] A human readable name for the system component.

  • description element (markup-multiline) [1] A description of the component, including information about its function.

  • purpose element (markup-line) [0 or 1] A summary of the technological or business purpose of the component.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

    Some specific property names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

  • annotation elements [0 to ∞] A name/value pair with optional explanatory remarks.

    Some specific annotation names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • status element [1] Describes the operational status of the system.

  • responsible-role elements [0 to ∞] Defines a role that has responsibility for the component.

    A role defined in this way can be associated with a party or parties here, or where the component is referenced (e.g., inventory-item, implemented-requirement/by-component).

    The roles "maintainer" or "provider" can be used to define the supplier. In such cases party-id(s) can be used to associate the providing organization. For products, services, and APIs, this is the vendor and/or retailer information. For policies, processes, plans, standards, and guidance, this is the authoring organization.

  • protocol elements [0 to ∞] Used for service components to define the protocols supported by the service.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

Components may be products, services, application programming interface (APIs), policies, processes, plans, guidance, standards, or other tangible items that enable security and/or privacy.

The type indicates which of these component types is represented.

When defining a service compoenent where are relationship to other components is known, one or more link entries with rel values of provided-by and used-by can be used to link to the specific component identifer(s) that provide and use the service respectively.

Remarks

A part provides for logical partitioning of prose, and can be thought of as a grouping structure (e.g., section). A part can have child parts allowing for arbitrary nesting of prose content (e.g., statement hierarchy). A part can contain prop objects that allow for enriching prose text with structured name/value information.

A part can be assigned an optional id, which allows for internal and external references to the textual concept contained within a part. A id provides a means for an OSCAL profile, or a higher layer OSCAL model to reference a specific part within a catalog. For example, an id can be used to reference or to make modifications to a control statement in a profile.

Use of part and prop provides for a wide degree of extensibility within the OSCAL catalog model. The optional ns provides a means to qualify a part's name, allowing for organization-specific vocabularies to be defined with clear semantics. Any organization that extends OSCAL in this way should consistently assign a ns value that represents the organization, making a given namespace qualified name unique to that organization. This allows the combination of ns and name to always be unique and unambiguous, even when mixed with extensions from other organizations. Each organization is responsible for governance of their own extensions, and is strongly encouraged to publish their extensions as standards to their user community. If no ns is provided, the name is expected to be in the "OSCAL" namespace.

To ensure a ns is unique to an organization and naming conflicts are avoided, a URI containing a DNS or other globally defined organization name should be used. For example, if FedRAMP and DoD both extend OSCAL, FedRAMP will use the ns "https://fedramp.gov", while DoD will use the ns "https://defense.gov" for any organization specific name.

Tools that process OSCAL content are not required to interpret unrecognized OSCAL extensions; however, OSCAL-compliant tools should not modify or remove unrecognized extensions, unless there is a compelling reason to do so, such as data sensitivity.

resource

Name Resource

Description A resource associated with the present document, which may be a pointer to other data or a citation.

This element appears inside: back-matter.

The resource element has the following contents (in order):

  • uuid attribute (uuid) [required] A RFC 4122 version 4 Universally Unique Identifier (UUID) for the containing object.

  • title element (markup-line) [0 or 1] A title for display and navigation

  • desc element (string) [0 or 1] A short textual description

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • doc-id elements (string) [0 to ∞] A document identifier qualified by an identifier type.

  • citation element [0 or 1] A citation consisting of end note text and optional structured bibliographic data.

  • rlink elements [0 to ∞] A pointer to an external copy of a document with optional hash for verification

  • base64 elements (base64Binary) [0 to ∞]

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

  • Any element (in a foreign namespace)

Remarks

Use of resource allows for information that pertains to, authenticates, or supplements the document it is included in.

A resource can be used in two ways. 1) it may point to an specific retrievable network resource using a rlink, and 2) be included as an attachment using a base64. A resource may contain multiple rlink and base64 entries that represent alternative download locations (rlink) and attachments (base64) for the same resource. Both rlink and base64 allow for a media-type to be specified, which is used to distiguish between different representations of the same resource (e.g., Microsoft Word, PDF). When multiple rlink and base64 entries are included for a given resource, all entries must contain equivalent information. This is extremely important when the entries represent OSCAL content that is represented in alternate formats (i.e., XML, JSON, YAML). A processor must be able to choose a preferred format to process (based on media-type) with the assumption that the same data can be processed from any of the available formats included by entries.

When a resource is used as a citation, then the title and citation properties must be included.

user

Name System User Class

Description A type of user that interacts with the system based on an associated role.

This element appears inside: local-definitions.

The user element has the following contents (in order):

  • uuid attribute (uuid) [required] The unique identifier for the user class.

  • title element (markup-line) [0 or 1] A title for display and navigation

  • short-name element (string) [0 or 1] A common name, short name or acronym

  • description element (markup-multiline) [0 or 1] A description supporting the parent item.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • annotation elements [0 to ∞] A name/value pair with optional explanatory remarks.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • role-id elements (NCName) [1 to ∞] A reference to the roles served by the user.

    The value may be locally defined, or one of the following:

    • asset-owner: Accountable for ensuring the asset is managed in accordance with organizational policies and procedures.
    • asset-administrator: Responsible for administering a set of assets.
    • soc: Members of the security operations center (SOC).
    • noc: Members of the network operations center (NOC).
    • incident-response: Members of the incident response team.
    • help-desk: Members of the help desk.
    • configuration-management-lead: Responsible for the configuration management processes governing changes to the asset.

  • authorized-privilege elements [0 to ∞] Identifies a specific system privilege held by the user, along with an associated description and/or rationale for the privilege.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

Permissible values to be determined closer to the application, such as by a receiving authority.
